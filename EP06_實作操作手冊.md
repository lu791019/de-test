### 先建立心智模型

EP05 我們是「直接呼叫 FinMind API」，一支程式從頭跑到尾。EP06 要把同一件事拆成三個角色：

```
Producer（發任務）  →  RabbitMQ（排隊）  →  Worker（執行）
producer.py            訊息佇列              celery worker
```

- **Producer**：只負責「把要做的事丟進佇列」，丟完就走，不等結果
- **RabbitMQ**：任務的候位區，任務在這裡排隊等人領
- **Worker**：從佇列領任務、真正去執行（呼叫 API、爬資料）

好處：Producer 跟 Worker 解耦，可以多開幾個 Worker 一起領任務平行處理。

## 第一部分：啟動 RabbitMQ（Docker Compose）

Worker 要連的「佇列」就是 RabbitMQ。先把它用 Docker Compose 拉起來。

### Step 1：啟動 Docker

```bash
sudo service docker start
sudo service docker status
```

✅ **預期**：`Active: active (running)`

### Step 2：用 rabbitmq.yml 啟動 RabbitMQ + Flower

```bash
cd ~/stock-crawler
docker compose -f rabbitmq.yml up -d
```

✅ **預期**：

```
[+] Running 3/3
 ✔ Network stock-crawler_dev       Created
 ✔ Container stock-crawler-rabbitmq-1  Started
 ✔ Container stock-crawler-flower-1    Started
```

> `rabbitmq.yml` 會建立一個叫 `dev` 的網路，並啟動兩個服務：
> - **rabbitmq**：佇列本體，AMQP 通訊埠 `5672`、管理介面 `15672`
> - **flower**：Celery 的監控 Web UI，埠 `5555`（第五部分會用到）

### Step 3：確認容器都起來了

```bash
docker compose -f rabbitmq.yml ps
```

✅ **預期**：rabbitmq 和 flower 兩個都是 `Up`。

### Step 4：確認 Web 介面

```bash
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:15672    # 預期：200（RabbitMQ 管理介面）
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:5555     # 預期：200（Flower）
```

打開瀏覽器進 RabbitMQ 管理介面確認：

- 網址：http://localhost:15672
- 帳號 / 密碼：`worker` / `worker`

✅ **預期**：登入後看到 Overview 頁面，目前 **Queues** 是空的（還沒有人發任務）。

## 第二部分：啟動 Celery Worker

RabbitMQ 起來了，現在啟動 Worker 來「領任務」。Worker 會一直掛在前景等任務，所以這個終端機之後要一直開著。

### Step 1：先看懂 Worker 怎麼連到 RabbitMQ

```bash
cat crawler/config.py
cat crawler/worker.py
```

重點：

| 檔案 | 重點 |
|------|------|
| `config.py` | 用 `os.environ.get(key, default)` 讀環境變數。`RABBITMQ_HOST` 預設 `127.0.0.1`（本機跑 worker 時連到 Docker 對外開放的 5672） |
| `worker.py` | 建立 Celery app，`broker=pyamqp://worker:worker@127.0.0.1:5672/`，並用 `include=[...]` 告訴 Celery 去哪些檔案找 task |

### Step 2：啟動 Worker

```bash
cd ~/stock-crawler
uv run python -m celery -A crawler.worker worker --loglevel=info
```

✅ **預期**：先看到 `config.py` 讀到的設定（loguru 印出），接著是 Celery 啟動橫幅：

```
[config]
.> app:         task:0x...
.> transport:   amqp://worker:**@127.0.0.1:5672//
.> results:     disabled://

[queues]
.> celery           exchange=celery(direct) key=celery

[tasks]
  . crawler.tasks.crawler
  . crawler.tasks_crawler_finmind.crawler_finmind
  . crawler.tasks_crawler_finmind.crawler_finmind_print

celery@你的主機名 ready.
```

看到 `ready.` = Worker 已連上 RabbitMQ，正在等任務。

> ⚠️ **這個終端機不要關**。Worker 是前景程序，關掉它就停止領任務了。發任務要「另外開一個終端機」。

## 第三部分：發第一個任務（print 版）

Worker 在等了，現在用 Producer 丟一個最簡單的任務給它。

### Step 1：開「第二個」終端機

保留第二部分的 Worker 終端機不動，**另外開一個新終端機**。

```bash
cd ~/stock-crawler
```

### Step 2：發任務

```bash
uv run python -c "from crawler.tasks import crawler; crawler.delay(x=0)"
```

> `crawler.delay(x=0)` = 把 `tasks.py` 裡的 `crawler` 任務丟進 RabbitMQ。
> `.delay()` 是「非同步」派送：呼叫完立刻回來，不等 Worker 跑完。
> （對比：直接寫 `crawler(x=0)` 是在本機同步執行，就失去分散式的意義了。）

✅ **預期**：指令秒回，沒有錯誤訊息。

### Step 3：回 Worker 終端機看結果

切回第一個（Worker）終端機。

✅ **預期**：看到任務被收到、執行、成功：

```
Task crawler.tasks.crawler[xxxxxxxx-...] received
crawler
upload db
Task crawler.tasks.crawler[xxxxxxxx-...] succeeded in 0.00s: 0
```

`crawler` 和 `upload db` 就是 `tasks.py` 裡那兩行 `print`。看到 `succeeded` = 完整跑通一輪「發任務 → 排隊 → Worker 執行」。

## 第四部分：把爬蟲包成 Celery task

print 版跑通了，現在換成真的爬蟲。EP05 我們直接呼叫 FinMind API，這次把它包成 Celery task，由 Worker 來執行。

### Step 1：看懂爬蟲 task

```bash
cat crawler/tasks_crawler_finmind.py
```

裡面有兩個 task：

| task | 行為 | 本集是否使用 |
|------|------|------------|
| `crawler_finmind_print` | 抓 FinMind 資料 → `print` DataFrame（**不寫 DB**） | ✅ 本集用這個 |
| `crawler_finmind` | 抓資料 → 寫入 MySQL | ❌ EP08 才用 |

本集還沒接資料庫，所以用 `crawler_finmind_print`。

### Step 2：看 Producer 怎麼批次發任務

```bash
cat crawler/producer_crawler_finmind_print.py
```

```python
from crawler.tasks_crawler_finmind import crawler_finmind_print

for stock_id in ["2330", "0050", "2317", "0056", "00713"]:
    print(stock_id)
    crawler_finmind_print.delay(stock_id=stock_id)
```

一個 for 迴圈，一次發 5 支股票的爬蟲任務。

### Step 3：發批次爬蟲任務

在「第二個」終端機（Producer 那個）執行：

```bash
uv run python crawler/producer_crawler_finmind_print.py
```

✅ **預期**：印出被派送的 5 個股票代碼：

```
2330
0050
2317
0056
00713
```

### Step 4：回 Worker 終端機看爬蟲結果

切回 Worker 終端機。

✅ **預期**：Worker 依序收到 5 個任務，每個都印出該股票的 FinMind 股價 DataFrame，最後 `succeeded`：

```
Task crawler.tasks_crawler_finmind.crawler_finmind_print[...] received
          date stock_id  Trading_Volume  ...   close  spread
0   2024-01-02     2330        ...        ...   593.0    ...
...
Task crawler.tasks_crawler_finmind.crawler_finmind_print[...] succeeded in 0.6s: None
```

看到 5 支股票的 DataFrame 都印出來 = 爬蟲已經成功被包成 Celery 任務、由 Worker 執行。

> 💡 一支 Worker 是「一次處理一個任務」。如果想要 5 支股票同時被多個 Worker 平行處理，可以再開幾個 Worker 終端機（都跑同樣的啟動指令），它們會各自從佇列領任務。

## 第五部分：用 Flower 監控任務

前面都是看終端機文字 log。Flower 提供一個 Web UI，可以視覺化看 Worker 狀態和任務數量。

Flower 在第一部分 `rabbitmq.yml` 就一起啟動了（埠 5555）。

### Step 1：打開 Flower

瀏覽器進入：http://localhost:5555

✅ **預期**：看到 Flower 介面，**Workers** 分頁列出你在第二部分啟動的 Worker（名稱類似 `celery@你的主機名`），狀態 **Online**，並顯示已處理（Processed）任務數量。

### Step 2：讓 Flower 也能看到「每一個任務」的明細

預設啟動的 Worker 不會送出「任務事件」，所以 Flower 的 **Tasks** 分頁可能是空的（只看得到 Worker 狀態）。要看到每個任務的明細，啟動 Worker 時加上 `-E`（送出事件）。

先回 Worker 終端機按 `Ctrl + C` 停掉，再重啟：

```bash
uv run python -m celery -A crawler.worker worker --loglevel=info -E
```

然後在第二個終端機重發一次任務：

```bash
uv run python crawler/producer_crawler_finmind_print.py
```

✅ **預期**：回 Flower 的 **Tasks** 分頁，看到 5 筆 `crawler_finmind_print` 任務，狀態由 `RECEIVED` → `STARTED` → `SUCCESS`。

> Flower 看的重點：
> - **Workers**：有幾個 Worker、在不在線、各自處理了多少任務
> - **Tasks**：每個任務的狀態（SUCCESS / FAILURE）、耗時、參數
> - 出問題時，FAILURE 的任務點進去可以看到 traceback，比翻終端機 log 快很多

## 第六部分：Docker 化 Worker

到目前為止 Worker 是用 `uv run` 跑在你的終端機（前景、要一直開著）。實務上 Worker 會被打包成 Docker 容器，在背景跑、可隨時水平擴展。這部分用 Docker Compose 把 Worker 容器化。

> ⚠️ **重要：網路要一致**
> 容器之間要互相溝通，必須在同一個 Docker 網路裡。
> - 前面用的 `rabbitmq.yml` 走的是自己建的 `dev` 網路。
> - 但 `docker-compose-worker.yml` 走的是外部共用網路 `my_network`。
>
> 所以這部分改用 `rabbitmq-network.yml`（同樣走 `my_network`），讓 RabbitMQ 跟 Worker 容器在同一個網路、能用服務名稱 `rabbitmq` 互相找到對方。

### Step 1：停掉前面的本機 Worker 和 rabbitmq.yml

先回 Worker 終端機按 `Ctrl + C` 停掉本機 Worker，再停掉 `dev` 網路那組 RabbitMQ（避免 5672 / 5555 埠衝突）：

```bash
cd ~/stock-crawler
docker compose -f rabbitmq.yml down
```

✅ **預期**：rabbitmq 和 flower 容器被移除。

### Step 2：建立共用網路 my_network

```bash
docker network create my_network
```

✅ **預期**：印出一串網路 ID。

> 如果出現 `network with name my_network already exists` 代表之前建過了，忽略即可。

### Step 3：用 my_network 啟動 RabbitMQ + Flower

```bash
docker compose -f rabbitmq-network.yml up -d
docker compose -f rabbitmq-network.yml ps
```

✅ **預期**：rabbitmq 和 flower 都 `Up`，且這次掛在 `my_network` 上。

### Step 4：啟動 Worker 容器

```bash
docker compose -f docker-compose-worker.yml up -d
```

✅ **預期**：

```
[+] Running 1/1
 ✔ Container stock-crawler-crawler_twse-1  Started
```

> 第一次會先 `Pull` 映像檔 `linsamtw/tibame_crawler:0.0.1`，稍等一下。

### Step 5：確認 Worker 容器有連上 RabbitMQ

```bash
docker compose -f docker-compose-worker.yml ps
docker compose -f docker-compose-worker.yml logs
```

✅ **預期**：容器狀態 `Up`，log 結尾看到 `celery@twse ready.`。

> 看一下 `docker-compose-worker.yml` 的 `command`：
> ```
> uv run python -m celery -A crawler.worker worker --loglevel=info --hostname=worker@%h -Q twse
> ```
> 這個容器 Worker 只監聽 **`twse`** 這個佇列（`-Q twse`），不是預設的 `celery` 佇列。
> 而且它用 `environment` 把 `RABBITMQ_HOST=rabbitmq` 覆蓋掉預設值，所以容器裡會連到同一個網路的 `rabbitmq` 服務。

### Step 6：發任務到 twse 佇列

因為容器 Worker 只聽 `twse` 佇列，我們要把任務「指定」送到 `twse`，否則任務會排在預設佇列沒人領。

在終端機執行（一樣用 print 版，不碰 DB）：

```bash
uv run python -c "from crawler.tasks_crawler_finmind import crawler_finmind_print; crawler_finmind_print.apply_async(kwargs={'stock_id': '2330'}, queue='twse')"
```

> `apply_async(kwargs={...}, queue='twse')` 跟 `.delay()` 一樣是派送任務，差別在可以指定 `queue`。
> 這正是「分流」的基礎：不同 Worker 聽不同佇列，各做各的（EP07 會深入）。

✅ **預期**：指令秒回，沒有錯誤。

### Step 7：看 Worker 容器的 log 確認執行成功

```bash
docker compose -f docker-compose-worker.yml logs --tail 30
```

✅ **預期**：看到 task received、印出 2330 的 DataFrame、最後 `succeeded`：

```
Task crawler.tasks_crawler_finmind.crawler_finmind_print[...] received
          date stock_id  ...   close  spread
0   2024-01-02     2330  ...   593.0    ...
Task crawler.tasks_crawler_finmind.crawler_finmind_print[...] succeeded in 0.7s: None
```

### Step 8：在 Flower 確認

瀏覽器進入 http://localhost:5555

✅ **預期**：**Workers** 分頁看到 `celery@twse` 在線（Online），Processed 數量增加。

### Step 9：收工清理

```bash
# 停 Worker 容器
docker compose -f docker-compose-worker.yml down

# 停 RabbitMQ + Flower
docker compose -f rabbitmq-network.yml down
```

✅ **預期**：所有容器被移除。`my_network` 可以留著，下次還能用。

## 常見問題（Troubleshooting）

| 症狀 | 原因 | 解法 |
|------|------|------|
| 發了任務 Worker 沒反應 | 任務送到的佇列沒有 Worker 在聽 | 本機 Worker 聽 `celery`（預設）；容器 Worker 聽 `twse`。用 `apply_async(queue=...)` 對應 |
| Worker 啟動報 `Connection refused` | RabbitMQ 還沒起來 / 沒連對 host | 確認 `docker compose ... ps` rabbitmq 是 Up；容器 Worker 要跟 RabbitMQ 在同一網路 |
| `docker compose -f rabbitmq-network.yml up` 報找不到網路 | 還沒建立 `my_network` | 先 `docker network create my_network` |
| 埠被占用（5672 / 5555） | `rabbitmq.yml` 跟 `rabbitmq-network.yml` 同時開 | 一次只開一組，切換前先 `down` |
| Flower 的 Tasks 分頁是空的 | Worker 沒送出事件 | 啟動 Worker 時加 `-E` |
