## 第一部分：config.py 環境變數實驗

### Step 1：看懂 config.py 的核心模式

```bash
cat crawler/config.py
```

每一個設定都是 `os.environ.get(key, default)`：

```python
import os

RABBITMQ_HOST = os.environ.get("RABBITMQ_HOST", "127.0.0.1")
RABBITMQ_PORT = int(os.environ.get("RABBITMQ_PORT", 5672))
MYSQL_HOST    = os.environ.get("MYSQL_HOST", "127.0.0.1")
MYSQL_PASSWORD = os.environ.get("MYSQL_PASSWORD", "ppWgnb_mfGe2m_")
```

> 💡 `os.environ.get(key, default)`：有設環境變數就用環境變數，沒設就用預設值。
> 開發時用預設值，部署時用環境變數覆蓋，**程式碼一行都不用改**。

### Step 2：不設環境變數（吃預設值）

```bash
cd ~/stock-crawler
uv run python -c "from crawler.config import RABBITMQ_HOST, MYSQL_HOST; print('RABBITMQ_HOST =', RABBITMQ_HOST); print('MYSQL_HOST =', MYSQL_HOST)"
```

✅ **預期**：

```
RABBITMQ_HOST = 127.0.0.1
MYSQL_HOST = 127.0.0.1
```

### Step 3：設環境變數（覆蓋預設值）

在指令前面臨時加環境變數，只對這一次執行生效：

```bash
RABBITMQ_HOST=rabbitmq MYSQL_HOST=mysql uv run python -c "from crawler.config import RABBITMQ_HOST, MYSQL_HOST; print('RABBITMQ_HOST =', RABBITMQ_HOST); print('MYSQL_HOST =', MYSQL_HOST)"
```

✅ **預期**：

```
RABBITMQ_HOST = rabbitmq
MYSQL_HOST = mysql
```

> 📌 同一份 `config.py`，因為環境變數不同，讀出來的值就不同。
> 這就是 Docker compose `environment:` 區塊能覆蓋設定的原理。

### Step 4：觀察 int() 轉型的必要性

```bash
RABBITMQ_PORT=5672 uv run python -c "from crawler.config import RABBITMQ_PORT; print(RABBITMQ_PORT, type(RABBITMQ_PORT))"
```

✅ **預期**：`5672 <class 'int'>`。

> 環境變數讀出來**都是字串**，所以 config 用 `int(os.environ.get("RABBITMQ_PORT", 5672))` 轉型，後續連線才不會型別錯誤。

## 第二部分：測試 retry 機制

我們寫一個「故意會失敗」的 task，觀察 Celery 自動重試。

### Step 1：建立一個測試 task

新增檔案 `crawler/tasks_retry_demo.py`：

```python
from crawler.worker import app


# bind=True 讓函式第一個參數是 self，才能呼叫 self.retry()
# max_retries=3：最多重試 3 次
@app.task(bind=True, max_retries=3)
def always_fail(self, x):
    try:
        print(f"嘗試處理 x={x}，第 {self.request.retries} 次")
        raise ValueError("故意失敗，模擬網路/API 錯誤")
    except Exception as exc:
        # 指數退避：第 0 次等 2 秒、第 1 次等 4 秒、第 2 次等 8 秒
        countdown = 2 ** self.request.retries
        print(f"失敗，{countdown} 秒後重試...")
        raise self.retry(exc=exc, countdown=countdown)
```

### Step 2：把這個 task 加進 worker 的 include

編輯 `crawler/worker.py`，在 `include=[...]` 清單加入新模組：

```python
app = Celery(
    "task",
    include=[
        "crawler.tasks",
        "crawler.tasks_crawler_finmind",
        "crawler.tasks_retry_demo",   # ← 新增這行
    ],
    broker=f"pyamqp://{WORKER_ACCOUNT}:{WORKER_PASSWORD}@{RABBITMQ_HOST}:{RABBITMQ_PORT}/",
)
```

> ⚠️ 沒加進 include，Worker 不會載入這個 task，發任務會一直 PENDING。

### Step 3：啟動 RabbitMQ + Worker（本機跑 Worker）

```bash
cd ~/stock-crawler
docker compose -f rabbitmq-network.yml up -d        # 需 my_network
uv run celery -A crawler.worker worker --loglevel=info
```

> 這裡用「本機」跑 Worker（不是容器），所以 `RABBITMQ_HOST` 吃預設 `127.0.0.1`，連 host 上的 RabbitMQ。
> 保持這個視窗開著，看 log。

### Step 4：開另一個終端機發任務

```bash
cd ~/stock-crawler
uv run python -c "from crawler.tasks_retry_demo import always_fail; always_fail.delay(x=1)"
```

### Step 5：觀察重試

回到 Worker 的視窗看 log。

✅ **預期**：看到類似輸出，間隔越來越久（2 → 4 → 8 秒）：

```
嘗試處理 x=1，第 0 次
失敗，1 秒後重試...
嘗試處理 x=1，第 1 次
失敗，2 秒後重試...
嘗試處理 x=1，第 2 次
失敗，4 秒後重試...
嘗試處理 x=1，第 3 次
... Task ... raised unexpected: MaxRetriesExceededError
```

打開 Flower（`http://localhost:5555`）→ Tasks，會看到該任務經歷 **RETRY** 狀態，最後變 **FAILURE**。

> 📌 重點：retry 撐過暫時性故障；用完 `max_retries` 才真的失敗。
> 真實的 `crawler_finmind` 也可以這樣包：成功就寫 DB，失敗就 `self.retry`。

### Step 6（觀念）：timeout 與 rate limit

裝飾器還能加這些保護（本集理解即可，不一定要實跑）：

```python
@app.task(
    bind=True, max_retries=3,
    time_limit=120,        # 硬逾時：超過 120 秒強制砍掉
    soft_time_limit=100,   # 軟逾時：100 秒先丟例外，讓你收尾
    rate_limit="10/m",     # 速率限制：每分鐘最多 10 次，當禮貌爬蟲
)
def crawler_finmind(self, stock_id):
    ...
```

| 機制 | 作用 |
|------|------|
| `time_limit` | 任務跑太久強制終止（最後防線） |
| `soft_time_limit` | 先丟 `SoftTimeLimitExceeded`，可 except 收尾 |
| `rate_limit` | 控制執行速率，避免被 API 擋（HTTP 429） |

## 第三部分：環境切換實作（local vs Docker）

同一份 `config.py`，怎麼做到「本機跑」和「Docker 容器內跑」都連對 RabbitMQ / MySQL？

### 為什麼容器內不能用 127.0.0.1？

容器內的 `127.0.0.1` 指的是**容器自己**，不是你的電腦。
所以容器要連 RabbitMQ / MySQL，得用 Docker 網路裡的**服務名稱** `rabbitmq` / `mysql`，
靠的就是大家都在 `my_network` 這張網路上。

### Step 1：本機模式（吃預設值 127.0.0.1）

第二部分我們就是用本機模式跑 Worker：

```bash
uv run celery -A crawler.worker worker --loglevel=info
```

啟動 log 會印出 `RABBITMQ_HOST: 127.0.0.1`。

### Step 2：Docker 模式（compose 覆蓋成服務名）

```bash
cat docker-compose-worker-network.yml
```

看 environment 區塊：

```yaml
services:
  crawler_twse:
    command: uv run celery -A crawler.worker worker --loglevel=info --hostname=%h -Q twse
    environment:
      - RABBITMQ_HOST=rabbitmq    # 覆蓋預設 127.0.0.1
      - MYSQL_HOST=mysql          # 覆蓋預設 127.0.0.1
    networks:
      - my_network
```

啟動容器版 Worker：

```bash
docker compose -f mysql.yml up -d                              # DB（爬蟲要寫入）
docker compose -f docker-compose-worker-network.yml up -d
docker compose -f docker-compose-worker-network.yml logs | tail -20
```

✅ **預期**：容器 Worker 的 log 印出 `RABBITMQ_HOST: rabbitmq`、`MYSQL_HOST: mysql`（不是 127.0.0.1）。

### Step 3：對照表

| 設定 | 本機（預設值） | Docker（compose 覆蓋） |
|------|--------------|----------------------|
| `RABBITMQ_HOST` | `127.0.0.1` | `rabbitmq` |
| `MYSQL_HOST` | `127.0.0.1` | `mysql` |
| 怎麼設定 | 不用設，吃預設值 | yml 的 `environment:` 區塊 |

> **一句話總結**：預設值給本地開發用，環境變數給部署環境覆蓋。
> 程式碼一行都不用改，換環境只換環境變數。這就是 12-Factor App 的 Config 原則。

## 第四部分：完整系統預演

把所有服務一起跑一遍，當作 EP12 整合的彩排。完整閉環 7 步驟，缺一不可。

### Step 1：啟動所有基礎設施

```bash
cd ~/stock-crawler
docker network create my_network          # 已建會說 already exists
docker compose -f rabbitmq-network.yml up -d
docker compose -f mysql.yml up -d
docker compose -f mysql.yml logs mysql | tail -5   # 等 ready for connections
```

### Step 2：確認所有 Web 介面

```bash
curl -s -o /dev/null -w "RabbitMQ   %{http_code}\n" http://localhost:15672
curl -s -o /dev/null -w "Flower     %{http_code}\n" http://localhost:5555
curl -s -o /dev/null -w "phpMyAdmin %{http_code}\n" http://localhost:8080
```

✅ **預期**：三個都回 `200`。

### Step 3：啟動多 Worker

```bash
docker compose -f docker-compose-worker-network.yml up -d
docker compose -f docker-compose-worker-network.yml logs | tail -30
```

✅ **預期**：`crawler_twse`、`crawler_tpex` 都 `ready`，環境變數顯示服務名。

### Step 4：發任務

```bash
docker compose -f docker-compose-producer-network.yml up
```

✅ **預期**：印出 `send task_2330 task` / `send task_00679b task`，容器 `Exited(0)`。

### Step 5：確認 Worker 執行成功

```bash
docker compose -f docker-compose-worker-network.yml logs | tail -40
```

✅ **預期**：看到 `crawler_finmind[...] succeeded`，不是 FAILURE。

### Step 6：Flower 確認

打開 `http://localhost:5555` → Tasks，2 筆 `crawler_finmind` 狀態 **SUCCESS**。

### Step 7：MySQL 驗證資料

```bash
uv run python -c "
import pandas as pd
from sqlalchemy import create_engine
from crawler.config import MYSQL_ACCOUNT, MYSQL_PASSWORD, MYSQL_HOST, MYSQL_PORT
engine = create_engine(f'mysql+pymysql://{MYSQL_ACCOUNT}:{MYSQL_PASSWORD}@{MYSQL_HOST}:{MYSQL_PORT}/mydb')
df = pd.read_sql('SELECT COUNT(*) AS cnt FROM TaiwanStockPrice', con=engine)
print('TaiwanStockPrice 筆數:', df['cnt'][0])
"
```

✅ **預期**：筆數 > 0，代表整條 pipeline 從頭到尾都通。

> 📌 端到端閉環的 7 步驗證：服務啟動 → Web 介面 → Worker ready → 發任務 → Worker succeeded → Flower SUCCESS → DB 有資料。
> **跑不完這 7 步，就不能說「系統正常」。**

### 收尾：關閉服務

```bash
docker compose -f docker-compose-worker-network.yml down
docker compose -f rabbitmq-network.yml down
docker compose -f mysql.yml down       # data volume 保留
```
